## 1. 배경 (Why)

온보딩에서 한 번 입력한 부모·자녀·관심사·알림 시간대는 다음 이유로 **언제든 수정 가능**해야 한다:

- **아이는 자란다** — 생년월일은 고정이지만 특이사항(알레르기·복용 약)은 시기마다 갱신, 자녀 추가/삭제 빈번.
- **사용자 관심사는 변한다** — 처음엔 "말문터지기"였다가 다음 분기엔 "사회성·또래관계"로 옮겨감. 콘텐츠 큐레이션의 정확도 직결.
- **알림은 라이프스타일 따라 조정** — 입사·이직·출산휴직 등 일과 변경 시 알림 시간을 새로 맞춰야 함. 또한 새 디자인(`2395:9126`)에서 **알림 종류가 2개로 분리**됨(10분 놀이 알림 + 주간 리포트 알림) — 종류별 enabled·시간 따로 관리 필요.
- **이용약관·동의는 법정 보존** — 마케팅 수신동의 변경 가능해야 GDPR/개인정보보호법 준수.
- **계정 탈퇴** — 사용자가 명시적 탈퇴(soft delete)할 수 있어야 함.

육아밸 v0.1 첫 진입 시 온보딩만 있었으나, 사용 시작 후 데이터 수정 경로 부재가 가장 큰 이탈 리스크. 본 기능으로 해소.

## 3. 도메인 영향

### 3.1 알림 모델 확장 — **핵심 변경**

현재 `User.notification` 단일 슬롯/시간(혹은 `NotificationPreference` 단일 row). Figma `2395:9126`은 **종류별** 알림 enabled + time:

| 종류                  | key             | 의미                                          | 기본 enabled |
| --------------------- | --------------- | --------------------------------------------- | ------------ |
| 10분 놀이 알림        | `play_10min`    | 미션 알림(기존 온보딩의 notification)         | true         |
| 주간 리포트 알림      | `weekly_report` | 주말 주간 리포트 발송 알림                    | true         |

**도메인 변경**:

| 엔티티                   | 변경 종류                                                                                            | 비고                                                       |
| ------------------------ | ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| `NotificationPreference` | 모델 신규 (또는 기존 단일 row → composite key `(userId, type)`)                                      | type enum: `play_10min` / `weekly_report`                  |
| `User.interests`         | 기존 `OnboardingDraft.interests` 영속화 위치 확정 (`User`의 1:N `UserInterest` 또는 `String[]`)      | /me 응답에 포함되어야 설정에서 hydrate                     |
| `Child`                  | (변경 없음) — 추가/수정/삭제 기존 모델 사용                                                          | soft delete 검토(이미 `deletedAt` 있음)                    |
| `User`                   | `deletedAt`, `deletionReason?`                                                                       | 계정 탈퇴 — cascade는 Prisma `onDelete: Cascade` 활용      |

> 도메인 변경이 있으므로 [`../schema/01-user.md`](../schema/01-user.md), [`../schema/`](../schema/) 갱신 동시 진행.

### 3.2 `/me` API 응답 확장

설정 화면 hydration을 위해 `/me`가 다음 필드를 추가 노출:

```ts
type Me = {
  id: string;
  onboardedAt: string | null;
  parent: { name; birthDate; gender; workStatus };
  interests: InterestId[];
  notificationPreferences: { type; enabled; time }[];
  children: { id; name; birthDate; gender; notes }[];
  consent: { service; privacy; marketing };
};
```

현재 `ServerMe`는 `{ id, onboardedAt }`만 — 큰 폭 확장 필요. 부분 확장(필요 화면 진입 시 별도 fetch)도 옵션 — 의문점 §7 참조.

### 4.1 `yougabell-api`

#### Prisma schema diff

```prisma
model NotificationPreference {
  id        String   @id @default(uuid()) @db.Uuid
  userId    String   @db.Uuid
  type      NotificationType
  enabled   Boolean  @default(true)
  time      String   // "HH:MM" (24h)
  updatedAt DateTime @updatedAt
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, type])
  @@index([userId])
}

enum NotificationType {
  play_10min
  weekly_report
}

model User {
  // ...기존 필드
  deletedAt        DateTime?
  deletionReason   String?
  notificationPreferences NotificationPreference[]
}
```

마이그레이션 이름: `add_notification_preferences_per_type` + `add_user_deletion`.

#### 엔드포인트

| 메서드 + 경로                          | 설명                                                                      | Auth      |
| -------------------------------------- | ------------------------------------------------------------------------- | --------- |
| `GET /me`                              | 확장된 사용자 + 자녀 + 관심사 + 알림 + 동의 일괄                          | JWT       |
| `PATCH /me/parent`                     | name·birthDate·gender·workStatus 부분 갱신                                | JWT       |
| `PATCH /me/interests`                  | `{ interests: InterestId[] }` (최대 3개 서버 검증)                        | JWT       |
| `PATCH /me/notifications/:type`        | `{ enabled, time }` upsert. type ∈ play_10min / weekly_report             | JWT       |
| `POST /me/children`                    | 자녀 추가 (Body: name/birthDate/gender/notes)                             | JWT       |
| `PATCH /me/children/:id`               | 자녀 수정                                                                 | JWT (소유) |
| `DELETE /me/children/:id`              | 자녀 삭제 (soft delete, deletedAt set)                                    | JWT (소유) |
| `DELETE /me`                           | 계정 탈퇴 (User.deletedAt + cascade. 토큰 무효화)                          | JWT       |

#### Auth Guard 분기

기존 `JwtAuthGuard` + 자녀 라우트는 `ChildOwnerGuard`(자녀가 요청자의 자녀인지 검증)로 추가 보호.

#### OpenAPI export

`@nestjs/swagger`로 자동 — `prisma:validate` 후 `pnpm openapi:export` 갱신.

## 6. 비기능 요구

### 저장 전략

- **즉시 서버 PATCH** — 온보딩과 달리 draft 누적 X. 각 필드/카드 단위로 변경 즉시 PATCH 호출 + 성공 토스트.
- **낙관적 업데이트 X** — 1차에선 모든 변경은 PATCH 응답을 기다린 후 UI 갱신(에러 복구 단순화). v2에서 SWR/TanStack Query 도입 시 낙관적 업데이트 검토.
- **멀티 디바이스 일관성**: web/mobile 동시 사용 시 PATCH 충돌은 last-write-wins. 알림 토픽은 native에서 한 번 더 sync.

### 보안

- 자녀 라우트(`/me/children/:id`)는 `ChildOwnerGuard`로 타 사용자 자녀 접근 차단.
- 계정 탈퇴는 **이중 확인** — 모달 1단(메인 화면) + 모달 2단(추후 비밀번호 또는 OTP 검토). 1차에선 단일 모달이지만 향후 강화.
- 마케팅 수신동의 변경 가능 시 변경 이력 `ConsentChangeLog` 검토(GDPR).

### 성능

- 메인 설정 진입 시 `/me` 1회 호출로 hydrate. sub 화면 진입 시 추가 fetch 없음(서버 컴포넌트 prefetch + 클라이언트 draft).
- 변경 PATCH 응답 200ms 이하 목표.

### 접근성

- 토글: `role="switch"` + `aria-checked`.
- 모든 row: 키보드 포커스 + Enter 키 trigger.
- 시간 휠: `role="listbox"`/`option` + aria-label(이미 온보딩에서 적용).
