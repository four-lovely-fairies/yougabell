# `docs/design/` 작성 규칙

> 워크스페이스·아키텍처·인프라의 **장기 결정 문서**. 자주 안 바뀌고, 바뀔 때마다 의도와 트레이드오프를 보존.
> 상위 룰: [`../AGENTS.md`](../AGENTS.md).

---

## 0. 다루는 것 / 다루지 않는 것

| 다룬다                                                    | 다루지 않는다                                 |
| --------------------------------------------------------- | --------------------------------------------- |
| 레포 분리 정책 / 모노레포 vs 폴리레포                     | 새 기능 기획 → `docs/features/`               |
| DB·ORM·인증·LLM Provider 선택                             | 도메인 엔티티 의미 → `docs/schema/`           |
| 호스팅 결정 (Vercel / Fly.io / EAS 등)                    | UI 컴포넌트·디자인 토큰 → 각 레포 `DESIGN.md` |
| 인프라 토폴로지 (CDN, Edge Function, Queue 등)            | 임시 작업 스크래치 (`docs/logs/` 권장)        |
| 보안·컴플라이언스 정책 (개인정보 마스킹, 감사 로그 룰 등) |                                               |

## 1. 명명·구조

- 파일명: `NN-<topic>.md` — 두 자리 일련번호 + kebab-case 토픽
  - 예: `00-repo-strategy.md`, `01-auth-strategy.md`, `02-llm-routing.md`
- 일련번호는 **결정 순서**대로 (시간순), 도중 삭제돼도 번호 재사용 X
- 토픽은 핵심 결정 영역 한두 단어

## 2. 본문 구조 (권장)

```markdown
# <Topic>

> 작성일: YYYY-MM-DD · 상태: `accepted` / `superseded` / `deprecated`

## 0. 결정 요약

| 항목   | 결정   | 비고 |
| ------ | ------ | ---- |
| <축 1> | <결정> | …    |

## 1. 컨텍스트 (왜 결정이 필요했는가)

…

## 2. 고려한 옵션

| 옵션 | 장점 | 단점 | 채택 여부 |
| ---- | ---- | ---- | --------- |

## 3. 선택과 근거

…

## 4. 영향 (어떤 레포·작업이 따라가야 하는가)

…

## 5. 후속 결정 / 대체 시점

…
```

## 3. 결정 변경 처리

- 새 결정으로 기존 문서가 **superseded**되면:
  - 기존 파일 헤더 상태를 `superseded by NN-<new-topic>`로 갱신 (삭제 X — 이력 보존)
  - 새 파일에서 "1. 컨텍스트"에 이전 결정 폐기 사유 명시
- 부분 수정은 동일 파일에 추가하고 변경일 명시

## 4. multi-repo 영향

설계 결정은 거의 항상 여러 레포에 영향. §4 "영향" 섹션에서:

- 어떤 레포가 어떤 변경을 받는지 명시
- 즉각 변경 vs 점진 변경 구분
- 영향이 큰 결정은 추가로 [`docs/features/`](../features/)에 마이그레이션 기획서 별도 작성 권장

## 5. 외부 자료

벤더 공식 문서(예: Vercel, Supabase) 링크는 **인용 시점의 URL**과 **확인 일자** 함께 — 외부 페이지가 옮겨가도 추적 가능.
